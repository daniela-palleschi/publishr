Code chunks

Structure

```{r}
rt <- c(312, 287, 445, 301, 389)
mean(rt)
```
[1] 346.8
  • Code chunks are fenced with ```{r} at the top and ``` at the bottom
  • Each chunk is executed in sequence when the document renders
    • Output is inserted at that position
    • Code must be written linearly: it must run from top to bottom

Languages

  • R: {r}
  • Python: {python}
  • Julia: {julia}
  • Observable JS: {ojs}
  • different languages can coexist in the same document, each runs in its own engine
  • use the reticulate package to share objects between R and Python chunks (and to render Python to PDF)

Chunk options

  • Options are set with the #| prefix directly under ```{r}
```{r}
#| echo: false
#| warning: false
5 * 6
```
Option Effect
echo show or hide code
output show or hide output
eval run or skip chunk
include include chunk and output
warning show or hide warnings
message show or hide messages

Global defaults

  • you can set these options once in YAML and override per chunk:
execute:
  echo: false      # hide code from output
  warning: false   # suppress warnings
  message: false   # suppress messages
  • to override for a specific chunk:
```{r}
#| echo: true
#| label: tbl-desc
#| tbl-cap: "Descriptive statistics."
tbl_desc |> gt()
```

Chunk labels

Every figure or table chunk should have a label with the correct prefix:

```{r}
#| label: fig-rt       # figures must start with fig-
#| label: tbl-desc     # tables must start with tbl-
```

Warning

Duplicate labels cause render errors. If you copy a chunk, always update the label.

Loading data

With here::here()

  • always use here::here() for project-relative file paths
    • portable across machines and operating systems:
df <- readr::read_csv(here::here("data", "raw", "your-data.csv"))

For processed data and saved model objects:

df_clean <- readRDS(here::here("data", "processed", "data_example.rds"))
fit_gaze       <- readRDS(here::here("output", "models", "fit_gaze.rds")) # we don't have this yet

Workflow tip

Load all data in a single setup chunk at the top of the document with #| output: false.

Saving processed data

Save processed objects so subsequent documents don’t need to re-run the full pipeline:

saveRDS(df_clean, here::here("data", "processed", "data_example.rds"))

Note

This is the link between your analysis scripts and your manuscript. The analysis script saves; the manuscript loads. The manuscript never re-runs the analysis.

Inline reporting

The idea

Instead of typing numbers manually:

“The sample comprised 48 participants.”

Compute and embed directly in prose:

The sample comprised ` r dplyr::n_distinct(df$px)` participants.

When data changes, re-render; the number updates automatically

Store values as objects

Compute once, use everywhere:

n_px  <- dplyr::n_distinct(df_clean$px)
n_obs <- nrow(df_clean |> filter(region == "verb"))
m_tt  <- round(mean(df_clean$tt, na.rm = TRUE), 1)
sd_tt <- round(sd(df_clean$tt,   na.rm = TRUE), 1)

Then in prose:

The sample comprised ` r n_px` participants (*N* observations = ` r n_obs`). Mean total reading time was ` r m_tt` ms (*SD* = ` r sd_tt`).

The sample comprised 24 participants (N observations = 1800). Mean total reading time was 350 ms (SD = 56).

APA 7 formatting

  • Italicise statistical symbols: *M*, *SD*, *t*, *p*
  • No zero before decimal for values that cannot exceed 1: *p* = .031 not *p* = 0.031
  • Round consistently: round(..., 1) for ms, round(..., 2) for correlations
fmt_p <- function(p) {
  ifelse(p < .001, "< .001", paste0("= ", round(p, 3)))
}
The effect was significant (*p* ` r fmt_p(p_value)`).
